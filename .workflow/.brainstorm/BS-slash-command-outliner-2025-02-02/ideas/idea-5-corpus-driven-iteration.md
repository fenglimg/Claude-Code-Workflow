# Deep Dive: Corpus-driven / 逐命令迭代校准（Golden 回归）

**Idea Score**: 9/10
**Created**: 2026-02-02T23:41+08:00
**Status**: Active - Proposed for cycle execution

---

## 概念

用现有全部 slash 命令作为“回归语料库（corpus）”，逐个命令进行：
**输入（规则文档/需求）→ 生成（大纲/文档包）→ 对比（现有实现）→ 产出 gap-report → 修复生成器 → 回归**。

它的价值是：把“生成器是否靠谱”的问题，从主观讨论变成可量化/可回归的工程问题。

---

## 关键原则：输入与评估分离（防信息泄漏）

- **输入**：仅允许使用该命令的规则文档（以及通用规范，如 SKILL-DESIGN-SPEC）。
- **评估**：使用现有实现作为 oracle（命令 MD + 相关脚本/模块/模板），做结构化差异对比，产出 gap-report。

> 这可以避免生成器“偷看答案”，同时也能反向暴露：哪些能力必须通过 ACE-tool/代码检索才能补齐，而不能只靠规则文档。

---

## 迭代单元（每个命令的三件套产物）

1) `generated-outline.md`
2) `gap-report.md`
3) `fix-plan.md`

其中 `gap-report` 建议按四大类归因，便于修复闭环：
- 模板缺口（缺少某类结构/章节/输出）
- 规则抽取缺口（规则文档信息映射失败）
- 匹配策略问题（参考命令/模式选错或不稳定）
- 规范校验问题（frontmatter/tools/sections 等不合规）

---

## 覆盖策略（先抽样、后全量）

第一轮先选“代表命令集”（每类 1-3 个）：
- analyze / plan / execute / brainstorm / issue / learn

生成器稳定后再扩展到全量命令（否则前期噪声太大、迭代效率会很差）。

---

## 完成标准（建议先锁一个 MVP 判定）

MVP 通过门槛可以先用“硬性合规 + 结构覆盖”两类指标：

- **硬性合规**：frontmatter 必填字段、allowed-tools、章节结构等 0 重大错误
- **结构覆盖**：关键流程节点（阶段/交互点/输出文件）在 outline 中可追踪

后续再引入更细的评分（相似度/可读性/解释性）。

---

## 风险与对策

- 风险：某些命令的“规则文档”不包含实现所需信息 → 对策：允许生成器在“匹配阶段”使用 ACE-tool 做受控检索，并在输出中标注 provenance（来自规则文档/来自参考命令/来自代码检索）。
- 风险：全量命令成本高 → 对策：先分层抽样 + 渐进扩容 + 自动化回归脚本。
